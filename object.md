## 2. 核心痛点与功能场景定义（产品需求文档）

请基于以下六大核心场景进行技术方案设计：

### 场景一：动态复杂度的 DDL (Deadline) 驱动引擎与排雷先锋
* **核心痛点（参考 AutoGPT/Devin 的不足）：** 大学生和科研人员的任务往往不是线性的。普通的日历只会到点提醒，导致“作业截止前一天才想起来，但这个作业需要跑两天模型”的惨剧。此外，很多时间被浪费在“配置环境依赖”上。
* **功能实现：** Agent 能够解析输入的任务大纲，评估任务的“算力消耗”、“时间复杂度”甚至“环境配置难度”。例如，如果任务是“完成多模态幻觉检测的实验验证并对比 LoRA/DPO 的效果”，系统会倒推评估这需要解决远程服务器依赖、模型权重下载以及漫长的训练时间，从而在截止日期前两周就开始强提醒，并主动提议：“是否需要我先在后台帮你将基础环境跑通？”；如果只是“阅读一篇短文”，则提前两天提醒。

### 场景二：跨端算力调度中枢 (The Remote Dispatcher)
* **核心痛点：** 科研工作者经常在本地轻量级设备（如 i5 笔记本的 WSL2 环境）与远程重型算力服务器之间切换。手动使用 FileZilla 或 rsync 同步代码、排除庞大的模型文件、管理复杂的 SSH 端口极度耗费精力。
* **功能实现：** Agent 需具备跨平台感知与调度能力。能够自动读取 SSH 配置（如通过特定端口 9002 连接），在后台静默处理大体积项目文件夹的增量同步（智能排除 `.bin` 等权重文件），并能接管和监控远程 tmux 终端中的训练进程。

### 场景三：上下文感知的赛博健康与心流守护者
* **核心痛点（参考 ActivityWatch 的局限）：** 沉浸式写代码或看论文时，生硬的番茄钟会严重打断心流；但久坐极易引发左侧下背部等区域的劳损。此外，休息回来后往往会出现“上下文遗忘”的断层感。
* **功能实现：** 结合系统状态监控（System Hooks）。当检测到本地环境（如 WSL2）中 CPU 正在满负荷编译，或 SSH 终端正在拉取庞大的数据集时，系统保持静默。当检测到键盘/鼠标输入频率下降、且距上次活动已超过特定时间，触发系统级原生弹窗：“当前模型权重已下载完毕。建议起身进行 3-5 分钟的跳绳有氧恢复，释放下背部压力”。**并且在用户重新落座触发键盘时，在终端打印一段“Welcome Back”的极简进度摘要。**

### 场景四：自动化报告与投影片流水线 (The Output Generator)
* **核心痛点（参考 Jupyter/LaTeX 的痛点）：** 实验做完了，但整理脏数据、画出高大上的 Matplotlib 图表、再排版成精美报告的过程令人绝望。LaTeX 编译环境极易崩溃。
* **功能实现：** 用户只需在特定目录丢入生数据（CSV/JSON）并口述需求。Agent 自动编写 Python 脚本清洗数据、生成图表，并利用 Typst（取代笨重的 LaTeX）瞬间编译出排版精美的 PDF 实验报告，且保持数据脚本与最终图表的溯源关联。

### 场景五：多模态文献大脑与前沿哨站 (The Researcher)
* **核心痛点（参考 ChatPDF/Zotero 的痛点）：** 每天 arXiv 上涌现海量论文，通用大模型极易产生幻觉，且面对包含复杂公式推导、双栏排版的长篇 PDF 时，传统的 RAG 会将图表与上下文割裂，丢失逻辑链。
* **功能实现：** 设定特定追踪锚点。面对用户上传的长篇文献，调用本地高精度多模态解析工具（如 MinerU）提取版面信息。不仅能回答“这篇论文在计算 AUC ROC 时的基线模型是什么”，还能跨越数十篇论文追踪某个特定评估指标的演进路线。还能联系 zotero 等热门软件。

### 场景六：逻辑陪练与启发式导师 (The Sparring Partner)
* **核心痛点（参考 Copilot/Aider 的副作用）：** 在死磕复杂逻辑或算法（如树的遍历、图的搜索等场景）时，AI 直接给出完整代码极其不利于学生构建底层思维体系。
* **功能实现：** 作为“苏格拉底式导师”，当进入该模式时，Agent 被强行拦截直接输出代码的权限。它只能通过提供时间/空间复杂度分析，或给出刁钻的边界测试用例（Edge Cases）来触发用户程序的崩溃，引导用户自行 debug。
